# 오늘의신기템 통일형 릴스 커버 설계

날짜: 2026-07-12  
상태: 사용자 승인됨

## 목적

Instagram 릴스 목록에서 모든 게시물이 하나의 계정처럼 보이도록 통일된 커버를 자동 생성한다. 상품이 분명하게 보이는 영상 프레임, 하단 그라데이션, 같은 위치의 두 줄 훅 문구를 고정 규칙으로 사용한다.

문구는 대본 첫 두 문장을 자동 사용하되 온라인 관리자에서 수정할 수 있다. 배경은 수집 단계에서 추출한 여섯 프레임 중 하나를 자동 추천하고 관리자가 다른 후보로 교체할 수 있다.

## 확정 선택

- 문구: 대본 또는 SRT의 첫 두 문장을 자동 사용, 관리자 수정 가능
- 프레임: 여섯 후보 중 자동 추천, 관리자 교체 가능
- 게시: 원본 영상 끝에 0.5초 커버 구간을 붙이고 Instagram `thumb_offset`으로 그 시점을 지정
- 원본: `final.mp4`는 수정하지 않고 게시용 `publish.mp4`를 별도 생성

## 접근법 비교

### 채택: 영상 끝에 커버 구간 삽입

릴스 시작 훅과 음성을 보존하면서 Graph API가 영상 내부 프레임을 커버로 선택할 수 있다. 커버가 영상 끝에 짧게 보이지만 시청 시작 이탈에는 영향을 주지 않는다.

### 제외: 영상 첫 장면에 커버 삽입

구현은 단순하지만 가장 중요한 첫 0.5초가 정지 화면이 되어 유지율에 불리하다.

### 제외: 커버 JPG를 Instagram 앱에서 수동 지정

자유도는 높지만 자동 게시 흐름이 끊기며 외부에서 운영하는 관리자 목적과 맞지 않는다.

## 시각 템플릿

- 캔버스: 1080×1920, JPEG 품질 92
- 배경: 선택 프레임을 비율 유지한 채 중앙 크롭해 전체 화면 채움
- 그라데이션: 화면 높이 52%부터 시작해 하단에서 약 90% 검정
- 텍스트 안전 영역: 중앙 3:4 크롭에서도 남는 하단 영역 안쪽
- 좌우 여백: 64px
- 문구 위치: 하단 UI 영역을 피한 y=1280~1510px
- 글꼴: Worker PC의 맑은 고딕 Bold, 없으면 사용 가능한 한글 Bold 폰트 탐색
- 첫 줄: 흰색, 58px Bold
- 두 번째 줄: 따뜻한 노란색 `#F5D26A`, 58px Bold
- 외곽선: 검정 2px와 약한 그림자로 밝은 배경에서도 가독성 확보
- 문장당 최대 두 시각 줄, 전체 네 줄을 넘으면 생성 거부 후 관리자 수정 요청
- 별도 로고, 스티커, 테두리는 사용하지 않아 상품과 훅에 집중

## 자동 문구 추출

우선순위:

1. `subs.srt`의 첫 두 자막 블록
2. `script.txt`의 첫 두 문장
3. 관리자 입력

문장은 공백을 보존하고 픽셀 폭 기준으로 줄바꿈한다. 자동 문구가 지나치게 길거나 파일이 없으면 추측하지 않고 `문구 확인 필요` 상태를 반환한다.

## 자동 프레임 추천

후보는 `ops/assets/<id>/frames/f01.jpg`부터 `f06.jpg`까지다.

각 프레임을 동일 크기로 정규화한 뒤 다음 점수를 계산한다.

- 검은 픽셀 비율: 많을수록 큰 감점
- 평균 밝기: 지나치게 어둡거나 과노출이면 감점
- 대비: 일정 수준의 대비가 있으면 가점
- 선명도: 에지 분산이 높으면 가점
- 중앙 정보량: 중앙 영역 에지와 색상 변화가 높으면 가점

추천 결과와 모든 후보 점수는 `cover.json`에 기록한다. 추천은 기본값일 뿐 관리자의 선택이 항상 우선한다.

## 로컬 산출물

상품 작업 폴더에 다음 파일을 둔다.

```text
ops/assets/<id>/
  frames/f01.jpg ... f06.jpg
  cover.json          선택 프레임, 문구, 점수, 템플릿 버전
  cover.jpg           최종 1080×1920 커버
  publish.mp4         final.mp4 + 끝의 0.5초 커버 구간
```

`cover.json` 예시:

```json
{
  "version": 1,
  "selectedFrame": 2,
  "recommendedFrame": 2,
  "line1": "열쇠고리인 줄 알았죠?",
  "line2": "진짜 찍히는 카메라예요.",
  "thumbOffsetMs": 14400,
  "scores": {"1": 0.21, "2": 0.86}
}
```

## CLI

새 명령:

```text
python scripts/make_cover.py <id>
python scripts/make_cover.py <id> --frame 2
python scripts/make_cover.py <id> --line1 "..." --line2 "..."
```

기본 실행은 문구 추출, 프레임 추천, 커버 생성, 게시용 영상 생성을 한 번에 수행한다. 재실행은 같은 입력에 같은 결과를 내며 `final.mp4`를 덮어쓰지 않는다.

## Supabase 데이터와 Storage

온라인 관리자에서 후보와 최종 커버를 볼 수 있도록 private `completed-assets` 버킷을 사용한다.

`assets.kind`에 다음 종류를 추가한다.

- `cover_candidate`: 여섯 후보 프레임
- `reel_cover`: 최종 커버 JPG
- 기존 `thumbnail`, `final_video` 유지

`assets`에는 `metadata jsonb`를 추가해 후보 번호, 추천 여부, 템플릿 버전을 기록한다. 최종 선택과 문구는 작업 payload 및 결과에 남기고, 완료 후 `cover.json`과 파이프라인 snapshot에도 투영한다.

Storage 경로:

```text
covers/<product-id>/candidate-01.jpg
covers/<product-id>/candidate-02.jpg
covers/<product-id>/cover.jpg
```

브라우저는 private 파일의 signed URL만 받아 표시한다. 원본·중간·게시용 영상은 기존 로컬 및 공개 호스팅 정책을 유지한다.

## Worker 작업

새 작업 유형 `generate_cover`를 추가한다.

payload:

```json
{
  "frame": 2,
  "line1": "열쇠고리인 줄 알았죠?",
  "line2": "진짜 찍히는 카메라예요."
}
```

처리 순서:

1. 허용된 상품 ID, 프레임 번호, 문구 길이를 검증
2. `make_cover.py` 실행
3. 후보 여섯 장과 `cover.jpg`를 Storage에 upsert
4. `assets` 행을 멱등 upsert
5. job result에 선택 프레임, 커버 경로, `thumbOffsetMs` 기록
6. 파이프라인 snapshot 동기화

Worker가 꺼져 있으면 작업은 `queued`로 남는다.

## 관리자 UI

상품 상세 drawer에 `릴스 커버` 섹션을 추가한다.

- 후보 여섯 장 그리드
- 자동 추천 후보에 `추천` 배지
- 현재 선택 후보에 명확한 테두리와 라디오 상태
- 첫 줄·두 번째 줄 입력
- 9:16 최종 커버 미리보기
- `커버 생성` 버튼
- 실행 중 진행률, 완료 커버, 오류와 재시도 안내

후보 이미지가 아직 업로드되지 않았으면 빈 버튼을 보여주지 않고 `Worker에서 프레임을 업로드해야 합니다`라는 다음 행동을 표시한다.

## Instagram 게시 연결

게시 스크립트는 `publish.mp4`와 `cover.json`이 있으면 이를 우선 사용한다.

- `publish.mp4`를 `site/media/<id>.mp4`로 호스팅
- `cover.json.thumbOffsetMs`를 컨테이너 생성의 `thumb_offset`에 전달
- 커버가 없으면 기존 자동 프레임 선택으로 폴백
- 현재 계산한 `thumb_ms`를 컨테이너 파라미터에 전달하지 않는 버그를 함께 수정
- dry-run에는 선택 영상과 커버 시점을 출력

Instagram 게시와 광고처럼 외부 부작용이 있는 단계는 기존 승인 경계를 유지한다.

## 오류 처리

- 후보가 없으면 `fetch_video.py` 실행 안내와 함께 종료
- 한글 Bold 폰트를 찾지 못하면 생성 실패로 처리하고 사용 가능한 경로를 안내
- 문구가 안전 영역을 넘으면 자동 축소하지 않고 수정 요청
- FFmpeg 실패 시 기존 `final.mp4`, `cover.jpg`를 보존
- Storage 업로드가 실패하면 로컬 산출물은 보존하고 job을 실패 처리
- `thumb_offset`은 게시 영상 길이 안에 있는지 검증

## 테스트

- SRT 첫 두 문장 추출
- script 문장 폴백과 길이 검증
- 검은 여백이 많은 후보 감점
- 추천 프레임과 관리자 override 우선순위
- 텍스트 줄바꿈과 안전 영역 검사
- FFmpeg 게시 영상 명령 생성
- `cover.json` 결정론적 직렬화
- Worker `generate_cover` 명령 허용 목록과 payload 검증
- migration의 작업·자산 종류와 metadata 계약
- 관리자 후보 선택·문구 수정·작업 enqueue 계약
- publish 스크립트가 `publish.mp4`와 `thumb_offset`을 실제 전달하는지 검증

## 완료 기준

1. 상품 2의 여섯 프레임에서 합리적인 추천 후보가 선택된다.
2. 승인된 템플릿으로 1080×1920 `cover.jpg`가 생성된다.
3. 대본 첫 두 문장이 자동 입력되고 관리자가 수정할 수 있다.
4. 관리자에서 여섯 후보와 최종 커버를 확인할 수 있다.
5. 원본 `final.mp4`가 보존되고 `publish.mp4`가 별도 생성된다.
6. Instagram 컨테이너에 실제 커버 시점이 전달된다.
7. Worker가 꺼져 있어도 요청이 보존된다.
8. 전체 테스트와 관리자 빌드가 통과한다.
