# 관리자 릴스 커버 적용 대기열 설계

## 배경

관리자 상품 상세에서 커버 후보를 누르면 미리보기만 바뀌고 최종 커버는 저장되지 않는다. 최종 저장을 담당하는 `커버 생성` 버튼은 로컬 Worker가 오프라인일 때 비활성화되어, 외부에서 관리자를 사용할 때 선택을 확정할 수 없다. 그 결과 사용자가 5번을 선택해도 게시 승인 화면과 로컬 게시 영상은 기존 6번 커버를 계속 사용한다.

## 목표

- 후보 선택은 즉시 미리보기에 반영하되 저장된 최종 커버와 구분한다.
- 사용자가 명시적으로 `선택 커버 적용`을 눌렀을 때만 작업을 만든다.
- Worker가 오프라인이어도 적용 요청을 Supabase 대기열에 저장한다.
- 커버 적용 작업이 끝나기 전에는 Instagram 게시 승인을 막는다.
- Worker가 적용을 완료하면 최종 JPG, 게시 영상, 관리자 메타데이터가 같은 프레임과 문구를 가리킨다.

## 결정한 UX

자동 저장은 사용하지 않는다. 여러 후보를 비교할 때마다 작업이 만들어지는 것을 피하기 위해 후보 클릭은 미리보기만 담당하고, 저장은 명시적인 버튼 한 번으로 확정한다.

최종 커버와 현재 입력이 다르면 편집 영역에 `저장되지 않은 변경` 상태를 표시하고 버튼 문구를 `선택 커버 적용`으로 보여준다. Worker가 오프라인이면 버튼은 활성 상태를 유지하며, 도움말은 `Worker가 켜지면 적용하도록 대기열에 저장합니다`로 바뀐다. 최종 커버와 입력이 같으면 버튼은 `적용됨`으로 비활성화한다.

적용 요청이 `queued`, `claimed`, `running` 상태일 때는 선택 프레임과 문구를 최신 작업 payload에서 복원해 새로고침 후에도 화면이 이전 최종 커버로 되돌아가지 않게 한다. 이 동안에는 추가 커버 요청과 게시 승인을 막고 상태를 각각 `적용 대기` 또는 `적용 중`으로 표시한다.

## 데이터 흐름

1. `CoverEditor`는 최종 `reel_cover` 메타데이터와 최신 활성 `generate_cover` 작업을 함께 받는다.
2. 사용자가 후보와 두 줄 문구를 편집하면 최종값 또는 활성 작업값과 비교해 변경 여부를 계산한다.
3. `선택 커버 적용`은 기존 `enqueueGenerateCover`를 호출해 `frame`, `line1`, `line2`를 Supabase `jobs.payload`에 저장한다. Worker 온라인 여부는 작업 생성 조건이 아니다.
4. Worker가 `generate_cover` 작업을 가져가 기존 `make_cover.py`를 실행하고, 여섯 후보와 최종 JPG를 Storage 및 `assets`에 다시 동기화한다.
5. 관리자 새로고침 후 최종 `reel_cover.metadata.selectedFrame`이 선택값으로 바뀌고 커버 적용 상태가 완료된다.

## 게시 안전장치

활성 `generate_cover` 작업이 있으면 클라이언트의 `getPublishButtonState`가 게시 버튼을 비활성화한다. 버튼 문구는 대기 중일 때 `커버 적용 후 게시 가능`, 실행 중일 때 `커버 적용 중`으로 구분한다.

클라이언트 우회나 동시 요청에도 게시가 먼저 실행되지 않도록 `approve_publish_reel` RPC에도 같은 검사를 추가한다. 동일 상품에 `queued`, `claimed`, `running` 상태의 `generate_cover` 작업이 있으면 승인 작업을 만들지 않고 명확한 오류를 반환한다. 완료·실패·취소된 작업은 게시를 막지 않는다.

## 오류 처리

- 커버 작업 생성 실패: 편집값을 화면에 유지하고 오류 배너를 표시한다.
- 커버 작업 실패: 기존 최종 커버는 유지하며 관련 작업 목록에서 실패 원인을 확인한다.
- Worker 오프라인: 오류가 아니라 정상 대기 상태로 취급한다.
- 게시 승인 경합: RPC가 활성 커버 작업을 감지해 게시 작업 생성을 거부한다.

## 테스트

- Worker 오프라인에서도 변경된 커버는 적용 요청이 가능하다.
- 최종값과 같은 입력은 중복 적용할 수 없다.
- 활성 커버 작업의 payload가 새로고침 후 편집값으로 복원된다.
- 활성 커버 작업이 있으면 게시 버튼이 비활성화된다.
- 실패·취소·완료된 커버 작업은 게시를 막지 않는다.
- RPC는 활성 커버 작업이 있을 때 게시 작업을 만들지 않는다.
- 기존 게시 승인, Worker 처리, 관리자 빌드 테스트가 모두 통과한다.

## 범위 밖

- 후보 클릭 즉시 자동 저장
- 브라우저에서 JPG 또는 MP4 직접 렌더링
- 여러 커버 작업의 동시 실행
- Instagram 게시 승인 자동화
